## 1. Introduction and Scope
This test plan outlines the testing strategy for "Micro-task 2.2: POPIA Notice Presentation & Consent Logging". The primary goal is to verify the AI-Verifiable Deliverable/Completion Criteria specified in the project plan. Testing will focus on the correct presentation of the POPIA notice, logging of user consent (simulated), and logging of the notice event.

## 2. Test Strategy
-   **Testing Levels:** Unit and Integration testing.
-   **Testing Types:** Functional testing, Negative testing, Database interaction testing.
-   **Approach:**
    -   Verify the existence and content of POPIA notice files in the correct directory.
    -   Test the core logic in `core_handler.py` for POPIA notice presentation under different user conditions (new user, existing user without consent).
    -   Verify the simulated consent mechanism, including the update of the `popia_consent_given` flag in the Supabase `users` table.
    -   Verify the logging of POPIA notice sending in the `message_logs` table.
    -   Mock external dependencies (like Supabase client for specific scenarios if needed, though direct interaction is preferred for validation where possible).
-   **Environment & Tools:** Pytest framework, Supabase (or mocked equivalent for specific unit tests).
-   **Clarification on POPIA notice file location:** The specification mentions `content/` directory, but existing files ([`data/message_templates/popia_notice_en.txt`](data/message_templates/popia_notice_en.txt), etc.) are in `data/message_templates/`. This test plan will assume `data/message_templates/` as the correct path based on existing project structure, unless otherwise directed. Tests will be designed to verify file existence at this path.
